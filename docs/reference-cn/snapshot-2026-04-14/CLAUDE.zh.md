# CLAUDE.md — Claude Code 项目级指令

本文件对所有在此目录运行的 Claude 会话生效，优先级高于默认行为。
本文件是**自演进**的——每轮工作结束时必须执行"经验沉淀循环"(见第 6 节)。

---

## 1. 预执行检查清单（开始任何非平凡工作前必须完成）

在开始写代码、建任务列表、或改文件**之前**，必须按顺序回答：

### A. 系统意图澄清
- 本次工作面对的是**什么类型的系统**？
  - 选项：① 工程改进（template/tool）② 架构设计（system）③ 问题修复（bug）④ 实验探索（spike）
- 系统的**边界**在哪里？（外部依赖、接入点、所有者）
- 系统的**非功能需求**（NFR）优先级？（正确性 vs 可用性 vs 扩展性 vs 成本）
- 本次授权的**范围**是什么？超出的部分是否需要先对齐？

**如果用户输入里没有明确这些，必须在回答的第一段主动提出澄清问题，不得假设。**

### B. 工作层级声明
每轮必须声明属于以下哪一层，并用对应的评审维度：

| 层级 | 触发条件 | 评审维度 |
|------|---------|---------|
| **架构层** | 新增子系统、跨组件边界变更、NFR 变化、对外接口 | 控制面/数据面、RBAC、可观测性、一致性、扩展性、审计 |
| **工程层** | 配置、工具链、脚手架、单一组件内的重构 | 正确性、可维护性、社区对齐、默认值安全 |
| **修复层** | 已知 bug 或不一致的消除 | 最小修改范围、回归风险 |
| **探索层** | 方案对比、可行性验证 | 结论可复现、证据清单 |

**不允许把架构层问题当作工程层来做**——这是本项目过往出现过的典型偏差。

### C. 成本预算
- 本轮预估工作量：小（<30 min）/ 中（<2 h）/ 大（需分多轮）
- 如果是大型工作，必须先交付蓝图/路线图给用户审核，再进入 TaskList 执行
- 大型工作应按轮次切分，避免单次耗尽限额

## 2. 方案先行（文档强制要求）

**每轮改动必须产生两类文件，缺一不可：**

### A. 方案/审查文档
- 位置：`docs/reviews/claude-review-round-N-YYYY-MM-DD.md`
- 时机：**改动开始前**创建
- **方案未获用户批准前不得进入实施**
- 必须包含：
  - 原始目的 / 本轮关注范围 / 本轮不展开 / 判断标准
  - 层级声明（架构层/工程层/修复层/探索层）
  - 当前状态评估（架构层必须有差距分析）
  - 决策理由（每一项改动的 why）
  - 分阶段路线图（架构层必须有）

### B. 变更日志
- 位置：`docs/reviews/round-N-change-log-YYYY-MM-DD.md`
- 时机：改动**全部完成后**创建
- 必须包含：
  - `参照:` 双向链接方案文档
  - 变更文件清单（表格：文件 | 变更类型 | 摘要）
  - 每项变更的意图
  - 本轮未做的事（边界）
  - 自检结果（语法、一致性、关键文件存在性）

**格式参照**：`docs/reviews/round-4-change-log-2026-04-09.md`
**落地判定**：方案文档 + 变更日志 + 自检通过，三者齐备才可宣称"已完成"

## 3. 执行阶段规则

### 任务管理
- 使用 TaskCreate / TaskUpdate 管理任务列表
- 开始一个任务前标记 `in_progress`；完成立即 `completed`
- 独立任务并行（单条消息多个工具调用）

### 自检闭环（每项改动后必须执行）
1. 实现 → 2. 语法/逻辑验证 → 3. 架构自审 → 4. 记录到 changelog

### 节奏控制
- 连续 5+ 任务后主动节奏放缓
- 触发限额后不得自动恢复——等用户确认再继续
- 大型工作拆多轮交付

---

## 4. 范围边界守则

- **不做未经授权的范围扩张**：即使发现额外问题，也应列入"下轮建议"，不在本轮实施
- **路线图先批后行**：3 项以上任务的计划必须先让用户过目（特别是架构层工作）
- **偏差即时反馈**：如果执行过程中发现原方案不可行，立即停止并报告，不要自行调整

## 6. 经验沉淀循环（CLAUDE.md 自演进机制）

**每轮工作结束前必须执行以下 3 步：**

### 步骤 1：提取本轮的反馈模式
识别以下信号：
- 用户**纠正**（"不是这样"、"停一下"、"不对"）
- 用户**追加要求**（"还要"、"另外"、"同时"）
- 用户**澄清隐性意图**（"其实是"、"我要的是"）
- 用户**质疑方向**（"够了吗"、"完善吗"、"还缺什么"）

### 步骤 2：归类为可沉淀规则
对每条反馈问：
- 这是**一次性**问题，还是**模式性**问题？
- 如果是模式性，应写入 CLAUDE.md 的哪一节？
- 现有规则是否已覆盖？如覆盖则说明执行时未遵守，需加强自检；如未覆盖则需新增规则

### 步骤 3：更新 CLAUDE.md 并记录到 changelog
- 新规则直接加入 CLAUDE.md 对应章节
- 在本轮 changelog 的末尾单列"本轮 CLAUDE.md 更新"小节
- 下一轮会话开始时，第一件事是复读 CLAUDE.md

**不允许"意识到问题但不沉淀"**——用户已经付出对话成本指出问题，不沉淀就等于让用户付出同样成本第二次。

## 8. 项目背景

本项目的**实际定位**（2026-04-14 澄清）：

> 基于 Ansible 的**多服务器管理控制系统**（不是单纯的开发模板）。

这意味着：
- 它不只是"一套 role 集合"，而是一个**控制平面 + 数据平面**的完整系统
- 需要考虑：控制器 HA、RBAC、审计、可观测性、事件驱动、多租户、DR、CI/CD 集成、服务目录等
- 教学价值保留，但**架构深度优先于表面覆盖**

详细规则见：`docs/reviews/review-iteration-charter.md`
