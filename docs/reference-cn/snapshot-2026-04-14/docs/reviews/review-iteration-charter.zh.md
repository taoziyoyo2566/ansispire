# Review Iteration Charter

日期: 2026-04-09
适用范围:
- `docs/reviews/` 下后续所有 Round 文档
- Codex / Claude 对本仓库的持续审查与修订讨论

## 原始目的

这个仓库的原始目的不是“做一个最强、最全、最生产化的 Ansible 仓库”，而是:

1. 做一个适合学习和理解 Ansible 的 demo 项目。
2. 覆盖核心功能、常见模式、官方推荐组织方式和社区常见实践。
3. 在保留教学价值的前提下，让默认主路径尽量自洽、可运行、可验证。

换句话说，后续迭代的目标是:
- 提升教学正确性
- 提升默认路径的一致性和可执行性
- 减少“看起来像最佳实践，实际跑不通/有误导”的内容

不是:
- 为了“更像生产”而无限引入复杂度
- 为了追求覆盖面而牺牲默认体验
- 在没有明确收益时把 demo 改造成企业内部平台模板

## 评审优先级

每轮优先按以下顺序评估问题:

1. 会直接失败的问题
2. 会误导学习者的问题
3. 默认路径与文档不一致的问题
4. 安全工作流不正确的问题
5. 工具链 / 测试链条不闭环的问题
6. 低优先级优化、美化、扩展性问题

## 平台支持策略

这个仓库的定位是“学习 Ansible 的 demo”，不是只面向单一 Linux 发行版的业务项目。

因此默认规则是:

1. 不得在未获得用户明确同意前，擅自把平台支持范围从“多发行版学习场景”收窄为单发行版或单家族。
2. 如果发现代码与支持声明不一致，优先方案不是立即删除某个平台，而是先提出“支持分层”方案。
3. 平台支持必须分成 3 层表述，避免把“可参考”“已适配”“已测试”混为一谈。

建议分层:

- Tier 1: 明确支持且持续测试
  - 至少包含一个 Debian 系平台
  - 至少包含一个 RHEL-compatible 平台
- Tier 2: 有代码骨架/有文档说明，但未纳入当前 CI 或 Molecule
  - 例如 Debian 其他版本、AlmaLinux/Rocky 的扩展版本
- Tier 3: 仅作为学习讨论对象，不承诺默认可运行
  - 例如极简 Linux、容器专用镜像、缺少 Python/systemd 的发行版

默认判断:

- “企业常见平台”不应只用 Ubuntu/Debian 代表
- “极简 Linux 存在”不等于必须纳入默认主路径支持
- 如果一个平台缺少测试，不代表要删除它；可以先降为 Tier 2

## 修复结论规则

后续所有 Round 文档中，不允许直接把“方向判断正确”写成“已修复”。

每个问题必须使用以下三档状态之一:

- 已落地
- 已部分落地
- 方向确认，尚未收口

判定为“已落地”前，必须同时满足:

1. 代码已修改
2. 与之相关的文档已同步
3. 相关支持声明已同步
4. 没有明显的自洽性冲突

如果以上 4 项缺一，则不得标记为“已落地”。

## 证据要求

每轮在宣称某项“已修复”或“已收窄支持范围”前，必须做最小证据核对。

至少检查以下对应项:

- 代码实现
- README / docs
- role meta / argument_specs
- preflight / runtime guard
- 测试或验证路径

建议写成简表:

| 项目 | 已核对文件 | 结论 |
|------|------------|------|
| 平台支持 | ... | 已同步 / 未同步 |
| 文档路径 | ... | 已同步 / 未同步 |
| EE/CI | ... | 自洽 / 不自洽 |

## 防止方向跑偏的强制问题

每轮在提出“大改方向”前，必须先回答:

1. 这是不是在修正错误，还是在重定义项目范围？
2. 如果是在重定义范围，是否已经得到用户明确授权？
3. 这个改动对“学习 Ansible”是增加清晰度，还是只是减少维护成本？
4. 是否存在更温和的做法，例如“支持分层”而不是“直接删除支持”？

## 固定边界

默认主路径指:
- `README.md`
- `ansible.cfg`
- `requirements.yml`
- `inventory/production`
- `playbooks/site.yml`
- `playbooks/rolling_update.yml`
- `playbooks/vault_demo.yml`
- `roles/`
- `molecule/`
- `execution-environment.yml`
- `.github/workflows/ci.yml`

教学扩展示例指:
- `examples/`
- 自定义插件示例
- 非默认运行路径下的演示文件

默认要求:
- 默认主路径应尽量可运行
- 教学扩展示例可以更宽松，但必须明确标注“示例/非默认执行路径”

## 每轮开始前必须写明的内容

每一轮文档开头必须先写下面 4 项:

1. 原始目的
2. 本轮关注范围
3. 本轮不展开的内容
4. 判断标准

建议模板:

```md
## 原始目的
本仓库用于学习 Ansible 功能、特性与最佳实践；本轮继续围绕“教学正确、默认路径可运行、文档与实现一致”推进。

## 本轮关注范围
- ...

## 本轮不展开
- ...

## 判断标准
- 是否减少误导
- 是否让默认路径更自洽
- 是否引入了不必要复杂度
```

## 每轮结束前必须回答的问题

1. 这轮修改是否更贴近原始目的？
2. 这轮是否把“教学示例”和“默认路径”分得更清楚？
3. 是否产生了新的文档漂移？
4. 是否引入了新的环境依赖？
5. 是否还有未关闭的前置矛盾？

## 文档强制要求（2026-04-14 新增）

**每轮改动必须产生两类文件，缺一不可**（此规则同步写入项目根目录 `CLAUDE.md`）：

### A. 方案/审查文档

位置: `docs/reviews/claude-review-round-N-YYYY-MM-DD.md`

时机: **改动开始前或开始时**创建（先写方案，再执行）

必须包含:
1. 原始目的
2. 本轮关注范围
3. 本轮不展开的内容
4. 判断标准
5. 每项改动的架构决策理由

### B. 变更日志

位置: `docs/reviews/round-N-change-log-YYYY-MM-DD.md`

时机: **改动全部完成后**创建

必须包含:
- `参照:` 字段链接对应的方案文档
- 变更文件清单（表格：文件 | 变更类型 | 摘要）
- 每类变更的意图说明
- 本轮明确未做的事（边界说明）
- 自检结果（语法、一致性、关键文件存在性）

### 落地判定标准

宣称某轮"已完成"前，必须同时满足：
1. 代码已修改且通过自检
2. 方案文档已创建
3. 变更日志已创建，并与方案文档双向链接
4. 变更日志中的自检结果无错误
