# Audit Relay Pagination & Backfill Plan (2026-04-16)

## Background & Motivation
The current audit relay (`controller/audit/relay.py`) polls the Semaphore `/api/events` endpoint. If the relay service crashes or is restarted and remains offline for a period where more events are generated than a single API response contains, those intermediate events are silently dropped. For a "multi-server management control system" (Ansispire), audit integrity is paramount. 

## Proposed Solution (Client-Side Pagination & Backfill)
1. **API Pagination**: Update the `fetch_events` function to recursively fetch older events as long as the oldest event in the current batch is newer than the saved `cursor.json`.
2. **Atomic Cursor Updates**: Ensure that the cursor is only updated *after* a batch of events has been successfully `forward`ed to the sink.
3. **Heartbeat Emission**: Introduce a periodic internal event to verify the connection is alive.

## Phased Implementation Plan
1. **Phase 1: API Discovery**: Confirm Semaphore pagination parameters.
2. **Phase 2: Refactor `fetch_events`**: Implement while-loop fetch.
3. **Phase 3: Robust `tick` Processing**: Atomic cursor per tick.
4. **Phase 4: Documentation**: Sync README and AI_WORKFLOW.
