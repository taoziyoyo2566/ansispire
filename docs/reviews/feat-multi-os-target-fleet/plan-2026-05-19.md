### D. Framework Best-Practice Pre-Check (workspace W-R18, MANDATORY)

**Question**: is per-OS-family `include_tasks` branching the Ansible-native way to multi-OS-baseline, or is there a higher-level abstraction we're missing?

**Sources consulted**:
1. Existing project pattern — `roles/infra_baseline/tasks/main.yml:19-78` already does `when: ansible_facts['os_family'] == "Debian"` block + `fail` stubs for RedHat / Alpine. **Direction matches** the standard Ansible pattern.
2. Upstream Ansible documentation (refresh check needed before Phase 2): `ansible.builtin.package` is OS-agnostic but cannot abstract per-OS-family **repo definition** (apt keyring vs dnf-plugins config-manager vs apk community repo). For docker install specifically, vendor-recommended path is OS-specific.
3. `geerlingguy.docker` Galaxy role pattern (reference: widely-used community role) — uses exactly the `tasks/{Debian,RedHat,Alpine}.yml` + `include_tasks: "{{ ansible_facts['os_family'] }}.yml"` pattern. **Confirms direction**.
4. Service-manager abstraction: `ansible.builtin.systemd` is **wrong for Alpine** (OpenRC, not systemd). Use `ansible.builtin.service` which auto-detects, OR `ansible.builtin.openrc` for explicit Alpine handling. **Direction-revision flag**: current `main.yml:81` uses `systemd:` for Docker service start — this will fail on Alpine. Must refactor to `service:` (auto-detect) OR move Docker service activation into each per-family include_tasks (cleaner: Debian = systemd, Alpine = openrc explicitly, RHEL = systemd).
5. **Alpine docker packaging**: `community` repo provides `docker` + `docker-cli-compose` (not the same package names as Debian's `docker-ce` + `docker-compose-plugin`). Repo needs explicit enable via `/etc/apk/repositories`. → Per-family include_tasks owns this detail.
6. **RHEL docker packaging**: Rocky/Alma 9 ship `podman` by default; Docker requires the Docker CE repo enabled via `dnf config-manager --add-repo https://download.docker.com/linux/rhel/docker-ce.repo`. Packages: `docker-ce`, `docker-ce-cli`, `containerd.io`, `docker-compose-plugin` (same names as Debian, different repo).

**Conclusion**: ✅ **Direction matches**. Per-OS-family `include_tasks` is the right pattern. **One revision required**: `Baseline | Ensure Docker service is running and enabled` (`main.yml:81`) must move from `systemd:` to `service:` (cross-OS) OR migrate into per-family blocks. Recommended: migrate into per-family blocks for clarity (Debian + RHEL = `systemd:`, Alpine = `service:` with `state: started, enabled: true` which OpenRC supports via `rc-update`).

## §1. Purpose / Scope / Judgment criteria

### Purpose
Extend the Ansispire control fleet to manage **4 VPS across 2 OS families** under one hub (`ans-hk01`). Replace the RHEL-family `fail` stub in `infra_baseline` with a real dnf-based implementation, onboard 4 VPS into `[targets_debian]` + `[targets_rhel]`. Outcome: from `ans-hk01`'s Semaphore, an operator can dispatch a Job Template that ansibles all 4 targets in one PLAY RECAP.

### Scope (what lands)
1. **Inventory** — 4 real VPS host entries in `inventory/hosts.ini` `[targets_debian]` + `[targets_rhel]` (alias + IP + `ansible_python_interpreter` pin per OS).
2. **`roles/infra_baseline/tasks/redhat.yml`** — dnf-based docker install for Rocky 9 + AlmaLinux 9; replace `main.yml:61-68` fail stub with `include_tasks`.
3. **`roles/infra_baseline/tasks/main.yml`** — refactor Docker service start to be OS-family-aware (move into per-family blocks; family-agnostic tail keeps user/group/sudo/authorized_key as today). The Alpine `fail` stub stays untouched (`main.yml:70-78`) — picked up in TASK-007.B.
4. **`playbooks/deploy_target.yml`** — new playbook that runs `infra_baseline` against `[targets:children]` (no hub role, no audit role).
5. **Makefile** — `target-deploy TARGET_NODE=<group>` wrapper (mirrors `hub-deploy` UX). `TARGET_NODE=debian|rhel|all` (no `alpine` option this round).
6. **Semaphore demo template** — one Job Template "Ping all targets" in `controller/semaphore/bootstrap.yml` (or follow-up to it) that runs `ansible all -i inventory/hosts.ini -m ping --limit targets`.
7. **Sync Guard** — `docs/reference/feature-map/INDEX.md` + new `docs/reference/feature-map/multi-os-fleet.md` + `CHANGELOG.md [Unreleased]` + this plan's changelog.

### Out of scope (explicit "NOT done")
- **Hub on the targets**: targets are pure data plane; do not deploy Semaphore on them.
- **EDA `target_group` rendering**: WU-6 N4, downstream-gated.
- **OS hardening beyond docker+user**: no CIS profile, no AIDE, no kernel sysctl tuning. (Could be a later TASK-009 if needed.)
- **Cross-OS firewall normalization**: `ufw` on Debian, `firewalld` on RHEL, `iptables` on Alpine — out of scope; targets stay on cloud-provider security groups.
- **Backup/snapshot of targets**: out of scope.
- **Decommissioning runbook**: out of scope; "add target" pattern only.

### Judgment criteria (what makes this PR mergeable)
A1. All 4 hosts converge to identical end-state under `ansible -m ping` + `ansible -m command -a 'docker --version'`.
A2. `ansible-playbook playbooks/deploy_target.yml --check --diff` (dry-run) reports 0 errors on all 4 hosts.
A3. Real `make target-deploy TARGET_NODE=all` PASSES — PLAY RECAP shows `ok=N changed=M failed=0` for each host.
A4. Re-running the same command immediately reports `changed=0` (idempotency on both Debian + RHEL families).
A5. `make verify` still green (no regressions in lint / syntax / tests / dry-run).
A6. Alpine work explicitly NOT done — `main.yml:70-78` stub unchanged. Tracked as TASK-007.B.
A7. Per-OS Molecule coverage NOT expanded in this round (existing 4 scenarios cover Debian only) — added to follow-up if needed.

## §3. Decision Rationale (per change)

### D1 — Why per-family `include_tasks` vs single `tasks/main.yml` with conditionals
The current `main.yml` already uses `when:` blocks per family. As blocks grow (each will be 8–12 tasks: prereqs / repo / GPG / install / service / verify), a 30+ task main.yml becomes unreadable. Convention follows `geerlingguy.docker`: `include_tasks: "{{ ansible_facts['os_family'] }}.yml"`. Each per-family file is then self-contained and reviewable in isolation.

### D2 — Why `dnf config-manager --add-repo` over hand-rolled `dnf_repository:`
RHEL/Rocky/Alma's `dnf-plugins-core` package provides `config-manager` which understands Docker's `.repo` file format directly. Less brittle than reconstructing `[docker-ce-stable]` section by hand. Idempotent (re-add is no-op).

### D3 — Why Alpine `apk add` from `community` repo vs the `edge` testing repo
Alpine 3.20's `community` repo ships docker 26.x which satisfies `infra_baseline_docker_min_version: "24.0.0"` (per `defaults/main.yml:5`). `edge` would give 27.x but bring in unstable kernel modules. Stick to `community/stable` for production targets.

### D4 — Why move Docker service start into per-family blocks
Alpine uses OpenRC, not systemd. `ansible.builtin.systemd` will fail on Alpine. Options:
- (a) Replace `systemd:` with `service:` (auto-detect) — works but loses explicit semantics.
- (b) Move into per-family blocks — Debian/RHEL use `systemd:`, Alpine uses `service:` (or explicit `openrc:` once `community.general.openrc` is confirmed available in our `requirements.yml`).

**Decision: (b)**. Each family knows its own service manager; main.yml stays focused on Python assert + user/group/sudo/authorized_key.

### D5 — Why introduce `vars/{RedHat,Alpine}.yml` instead of overriding everything inline
`infra_baseline_docker_repo_url` is currently `"https://download.docker.com/linux/{{ ansible_facts['distribution'] | lower }}"`. This works for `ubuntu`/`debian`, but Alpine's docker doesn't ship from `download.docker.com/linux/alpine` (it comes from Alpine's own `community` repo). Per-family `vars/<OS_family>.yml` lets each family declare what it actually consumes without polluting `defaults/main.yml` with conditionals.

### D6 — Why `deploy_target.yml` is a new playbook vs reusing `site.yml` with `--limit targets`
`site.yml` runs the full stack (hub + audit). Targets must NOT receive that. A separate `deploy_target.yml` with `hosts: targets:children` + `roles: [infra_baseline]` makes the intent explicit and prevents accidental hub installation on a managed node. Mirrors `deploy_hub.yml`'s pattern.

### D7 — Why Semaphore "Ping all targets" template is in scope vs deferred
A1+A4 (judgment criteria) require operator-visible proof the fleet is reachable from the hub. The Semaphore template is the simplest, lowest-cost UI surface to demonstrate this. Without it, "TASK-007 done" lacks a one-click verification path. Cost: ~30 lines in `bootstrap.yml`; well below worth-it threshold.

### D8 — Why RHEL Python 3.9 → install python3.11 before assert
Probe (2026-05-19) showed `rocky9` + `alma9` ship Python 3.9.25 (default `/usr/bin/python3`) — below `infra_baseline_python_min_version: "3.10"` declared in `roles/infra_baseline/defaults/main.yml:4`. The current `main.yml:13-17` `assert Python ≥3.10` runs BEFORE any per-family `include_tasks`, so it would fail on RHEL 9 before redhat.yml gets a chance to install a newer Python.

**Decision**: refactor task order in `main.yml`:
- Move the Python assert from position 1 (line 13-17) to AFTER the per-family blocks.
- `redhat.yml` first task: `dnf install -y python3.11 python3.11-pip` (RHEL 9 community offers 3.11 in AppStream).
- `redhat.yml` second task: `set_fact: ansible_python_interpreter=/usr/bin/python3.11` + `meta: clear_facts` to re-detect.
- Then the family-agnostic Python assert runs against the just-installed 3.11.

**Alternative considered (rejected)**: lowering `infra_baseline_python_min_version` to `3.9`. Rejected because (a) 3.9 is EOL in 2025-10, (b) the project already runs 3.12/3.13 elsewhere, (c) lowering the floor weakens the contract for all future targets.

**Risk**: Inventory must pin `ansible_python_interpreter=/usr/bin/python3` for the FIRST run (to use the only Python that exists pre-install). After first run, can switch to `/usr/bin/python3.11` or keep `/usr/bin/python3` as a symlink swap. Plan §8.2 callout covers this state machine.

### D9 — Why `[hub_remote]` is left untouched in TASK-007 (out of scope)
Probe (2026-05-19) confirmed the previous remote hub `ans-hk01` (89.185.26.211:1156) is dead — the user reinstalled the OS, wiping the Semaphore + audit stack that was there. The same IP is now exclusively the Debian 13 target `d13` (port 22, user `root`).

TASK-007 only adds entries to `[targets_*]`; it does NOT touch `[hub_remote]`. The dead `ans-hk01` reference is a separate concern, tracked as a follow-up in `TODO.md` under "Stage 2 maintenance". Conflating cleanup with this fleet-expansion task would violate W-R13 minimum-modification.
