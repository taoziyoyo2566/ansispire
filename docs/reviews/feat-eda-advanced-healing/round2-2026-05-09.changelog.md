## 0. Method change this round (per user feedback)

User feedback at start of round 2: *"是否应该先做一个测试文档，可以清晰的做出可以使用的命令操作，以及预测结果。然后用实际结果对比来确定代码是否有问题。"*

This is the spec-first interpretation of `ansispire/CLAUDE.md` §3 ("TSVS Mandatory") that I had been doing the lazy version of (write tests, then file the TSVS). Round 2 reorders to **spec → code → run → backfill §6 actuals**, for every layer:

1. Author the TSVS doc with §1–§5 (intent, environment, methodology, expected results) **before** writing the test code.
2. Implement the test code to satisfy the spec.
3. Run; capture actual output.
4. Diff expected vs actual into TSVS §6; fill in §7 conclusions.

This makes "what was supposed to happen" auditable independently of "what did happen", and removes the confirmation-bias risk of writing both at the same time.
