**Correction following user feedback at round 1 close (CLAUDE.md granularity)**:

I initially proposed adding two rules to ansispire `CLAUDE.md` §3 — "+300 host port convention" and "verify against `operations.md`, not inferred steps". User pushed back: those are situational details — the port rule only matters when designing/editing port-related config, the verification rule only matters during validation of an existing module. Carrying them as resident context for every future session inflates token cost and dilutes attention with no benefit.

Reverting that proposal. CLAUDE.md is for **methodology** (how to think / how to work / priority), not for **configuration facts** (which port, which version, which file path) or **situational triggers** (when feature X exists, verify via Y). Configuration facts already live in their config files (`config/ports.yml`); situational verification rules already live in `docs/TESTING_GOVERNANCE.md`. No duplication.

Plan §4 Phase 4 updated accordingly: removed the CLAUDE.md edit; added an end-user **operator guide** as a deliverable distinct from `operations.md` — the latter is a maintainer-side quick reference, the former is what lets a user `git clone` and reach a working self-healing chain without an AI in the loop.

The general lesson — *CLAUDE.md = methodology, config files = facts; before adding to CLAUDE.md, ask "does this apply regardless of what I'm doing?"* — is itself methodology and applies to every project, so it is being captured in `~/workspace/CLAUDE.md` §8 as W-R14, not in this project's CLAUDE.md.
