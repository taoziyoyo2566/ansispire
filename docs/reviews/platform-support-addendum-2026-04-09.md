# Platform Support Addendum

日期: 2026-04-09
关联: [Review Iteration Charter](./review-iteration-charter.md)

## 原始目的

本仓库用于学习 Ansible 的能力边界、设计方式和实践差异，因此平台策略应体现“异构环境管理”这一现实，而不是过早退化成单发行版示例。

## 结论

把仓库简单收窄到 Debian/Ubuntu，不是当前阶段最好的处理方式。

更合适的做法是:

1. 明确“支持分层”
2. 在默认测试里保留一个 Debian 系平台和一个 RHEL-compatible 平台
3. 对极简 Linux 单独降级表述，不混进默认主路径承诺

## 推荐策略

### Tier 1: 默认支持并持续验证

- Ubuntu
- 一个 RHEL-compatible 发行版
  - 推荐 `Rocky Linux` 或 `AlmaLinux`

说明:
- 这样既能覆盖 Debian 系习惯，也能覆盖企业中常见的 RHEL 系习惯
- 对学习 Ansible 来说，这比只保留 Ubuntu 更接近真实使用场景

### Tier 2: 有骨架但暂不持续验证

- Debian 其他版本
- 其他 EL-compatible 版本

说明:
- 可保留 `vars/os/` 骨架
- 可在 README 中说明“存在适配入口，但当前未纳入 CI/Molecule”

### Tier 3: 仅作讨论，不承诺默认可运行

- 极简 Linux
- 缺少 Python 的主机
- 容器专用或裁剪极多的镜像

说明:
- 可以在 `examples/` 或文档中展示 bootstrap 思路
- 不应把它们纳入默认 role 行为承诺

## 术语说明

这里的几类目标，不是“不能支持”，而是“不能和常规 Linux 主机放在同一默认假设里”。

### 极简 Linux

通常指系统安装非常精简，只保留最少运行组件的环境，例如:
- 最小安装的企业 Linux
- 只带基础包管理器和 SSH 的云镜像
- Alpine 一类偏轻量发行版

这类环境常见问题:
- 默认缺少一些常用系统命令
- 服务管理器不一定是 `systemd`
- 包名、路径、网络工具与 Debian / RHEL 常规服务器不同

结论:
- 这类系统通常需要先检查环境依赖
- 但“需要补依赖”不等于“应该从支持矩阵里删掉”

### 无 Python 镜像 / 主机

Ansible 大多数模块依赖目标机上有可用的 Python 解释器。

“无 Python”常见于:
- 新建的极简云主机
- 很瘦的容器镜像
- 安全加固后只保留极少运行环境的机器

对这类目标，常见流程是:

1. 先用 `raw` 或 `script` 做 bootstrap
2. 安装 Python
3. 再切回正常的 Ansible 模块

因此它属于:
- 有明确处理方式
- 但不适合作为默认 role 前提

### 强裁剪系统

指为了体积、安全面或特定用途，对系统能力做了明显删除的环境，例如:
- 去掉包管理器缓存和大量用户空间工具
- 去掉编译工具、cron、syslog、firewall 工具
- 只保留单一业务进程的容器/镜像

这种环境下，很多 role 默认假设会失效，例如:
- 你以为有 `cron`
- 你以为有 `systemctl`
- 你以为有 `ufw` / `firewalld`
- 你以为可以直接安装额外包

正确处理方式是:
- 先检测依赖
- 再决定补齐依赖、降级功能，还是标注为非默认支持

而不是:
- 看到依赖缺失就直接缩小整个项目的平台范围

## 对“环境依赖检查”的统一原则

你提出的方向是对的:
- 遇到特殊平台，通常先检查环境依赖，并在必要时解决依赖

后续统一按下面顺序处理:

1. 先判断缺的是“业务依赖”还是“平台基础依赖”
2. 能通过 preflight 明确检测的，就先检测
3. 能通过 bootstrap 补齐的，就单独写 bootstrap / examples
4. 只有当补齐成本明显过高，且偏离仓库原始目的时，才考虑降级到 Tier 2 / Tier 3

这条规则适用于:
- 极简 Linux
- 无 Python 主机
- 强裁剪镜像
- 其他与默认假设不一致的系统

## 对 Claude 后续处理的约束

后续如果 Claude 再遇到“代码没有真正跨平台适配”的情况，默认动作应是:

1. 先标出“声明与实现不一致”
2. 再提出“支持分层”
3. 最后再问是否要正式缩减支持范围

而不是:

1. 直接把平台范围删掉
2. 再把文档和 meta 跟着改窄

## 后续建议

下一轮如果要收口平台问题，建议按这个顺序:

1. 先定义支持矩阵
2. 再同步 `README`、`meta`、`preflight`
3. 再决定 Molecule / CI 至少覆盖哪两个平台
4. 最后才处理是否把某些平台降级到 Tier 2
