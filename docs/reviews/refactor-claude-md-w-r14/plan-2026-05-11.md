## 3. Decision rationale

### Why this is a refactor, not a deletion
Every removed rule still binds. W-R14 is about *where the rule lives*, not whether it applies. The carriers cited above are authoritative homes that the affected workflow already reads (testing-governance.md is read when working on tests; ai-workflow.md is the AI-onboarding doc; INDEX.md is the entry point for any RCA).

### Why §0.4 stays despite mentioning a specific file
Sync Guard items are closure-time entry points. The pattern is universal ("at task closure, sync each truth source the change touched"); the file paths are pointers to those truth sources, not situational triggers. Same status as §0.1–§0.3.

The user-visible-vs-internal **trigger criteria**, however, is a CHANGELOG-policy detail and moves into `CHANGELOG.md`'s self-description.

### Why §3 keeps Evidence-based Verification but not TSVS
"Every change must be backed by terminal logs" is universal. "Functional tests must register in `docs/reference/test-specs/`" only fires when adding a functional test — that's situational AND already fully specified in `testing-governance.md §7`.

### Why §4 disappears entirely
Pattern A is workspace W-R10 verbatim. Pattern B is path-naming for one specific document family — belongs with that family.

### Why §5 grows
Adding the W-R14 audit principle as a meta-rule prevents future regressions. It IS universal: "before adding any rule to this file, run the W-R14 test." Applies to every CLAUDE.md edit.
