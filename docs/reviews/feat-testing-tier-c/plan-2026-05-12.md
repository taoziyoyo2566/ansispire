## 4. Judgment criteria (acceptance bar per item)

Each item is "done" only when:

1. **Idempotence proven**: `molecule converge && molecule verify && molecule verify` passes with `changed=0` on the second verify (or the equivalent for non-Molecule items).
2. **TSVS registered**: any new scenario gets a `docs/reference/test-specs/molecule-<role>.md` entry following the existing template; `INDEX.md` table row added.
3. **CI matrix updated**: any new scenario lands in `.github/workflows/ci.yml` `molecule` job matrix; comment line stays accurate (lesson from 2026-05-12 `ci.yml:96` stale-comment fix).
4. **Local time budget honored**: per scenario, total `molecule test` runtime ≤ 5 min on the workstation; if exceeded, split or scope back.
5. **No regression** in existing `make verify-full` chain — full chain passes after each round's change.
6. **TSVS verification recipe runs clean**: the literal commands cited in the TSVS file execute (lesson from operator-guide R3 hotfix — verify executable content against source of truth).

## 6. Decision rationale (per item)

| Item | Decision | Rationale |
|---|---|---|
| G1 | New Molecule scenario `molecule/infra_baseline/` modeled after `molecule/common/` | `infra_baseline` is closest in role-shape to `common` (baseline-style OS prep + apt + docker install). Template lift-and-adapt keeps cost low. |
| G2 | **Recommend Molecule scenario over e2e expansion**. e2e (`controller/audit/e2e/run.sh`) already covers full hub deploy at L4 (~60 s); a Molecule scenario at L3-ish gives faster + isolated regression coverage for the rsync exclude / state-file invariants specifically. If the role's dependency on a docker engine in the container proves brittle, fall back to extending e2e checks. | The 21-pattern rsync exclude list in `roles/ansispire_hub/tasks/main.yml` and the `/var/lib/ansispire/state/` placement are exactly the invariants regressing under future refactor. Need a fast feedback loop, not just 60 s e2e. `ansispire_hub` itself only uses `community.docker.docker_compose_v2` (manages already-installed docker); the docker engine install lives in `infra_baseline`. G1's Molecule scenario will establish whether the privileged-systemd container pattern can host a real docker engine; G2 inherits that answer. |
| G3 | Add Debian 12 `geerlingguy/docker-debian12-ansible:latest` platform to existing `molecule/webserver/` + `molecule/database/` `molecule.yml`. Same template as `molecule/common/`'s Debian 12 platform block. | Lowest-cost cross-distro expansion. Same image already pre-built by upstream; just a copy-paste of the existing common scenario's Debian platform stanza. |
| G6 | `make verify-mid` reads `git diff --name-only origin/dev...HEAD`, maps changed paths to affected Molecule scenarios via a static table, runs only those. Falls back to `verify-full` if mapping fails. | The L2 piece. Path-table driven keeps the rule explicit + auditable; no guessing. Reuses existing per-scenario `molecule test -s <name>` invocation. |
| G7 | Parallelize via background-and-wait pattern in `Makefile`: a new `molecule-all-parallel` target that backgrounds `molecule test -s <each>` and waits, with per-scenario log files for post-hoc inspection. Existing serial `molecule-all` stays as default (CI parallelism is independent). | Make's native parallel mode (`-j`) does not work cleanly with Molecule's compose / network state — explicit background-wait avoids the foot-gun. |
| T-C1 | Convert `failed_when: false` on the `my.cnf` content check to a real hard assert. | Soft check provides no signal; remove the silent ratchet. |
| T-C2 | Drop the second `database__mysql_root_password:` from each affected `molecule.yml`; keep the first. | Yaml's last-key-wins behaviour means the current code is correct by accident; remove the read-time ambiguity. |
| T-C3 | Add a verify step that exec's `mysql -u testuser -p<pw> -e 'SELECT 1' testdb` from inside the container. | The current Ansible task that grants `testuser` returns ok without proving the grant works; the probe closes that gap. |

## 8. Risk / NFR

- **CI runtime budget**: adding G1 + G2 expands the Molecule matrix from 4 to 6 scenarios. Each runs in parallel on its own runner per the existing matrix; net wall-clock impact ≈ zero, but runner minutes ~+50%. Acceptable.
- **Local runtime budget**: serial `molecule-all` already at 10–20 min; adding 2 scenarios pushes to ~15–30 min serial. G7 parallel target offsets this. Until G7 lands, contributors should use `make verify-mid` (R5) or run scenarios individually.
- **dind feasibility** (G1 first, then G2): `infra_baseline` installs `docker-ce` + `docker-ce-cli` (verified at `roles/infra_baseline/tasks/main.yml:53`). Running that inside a Molecule container = docker-in-docker. The existing Molecule scenarios (common / webserver / database / full-stack) all use the privileged-systemd container pattern (`/sys/fs/cgroup` mount + `command: /lib/systemd/systemd`), which the upstream `geerlingguy.docker` role's own CI uses to install dockerd successfully. Expectation: G1 works under the same pattern. If R3 finds it does not, R3 reports back and re-plans; G2 (R4) inherits the answer either way. |
- **G6 path mapping correctness**: a wrong mapping silently skips scenarios. R5 includes a smoke test: deliberately mutate each role's `defaults/` and confirm `make verify-mid` picks up the right scenario.
