## 2. Intent per change

### T-C1 (my.cnf hard assert)

The original task ran `grep -q "Ansible managed" ...` with `failed_when: false` AND `register: mycnf_check`, but no downstream task ever inspected `mycnf_check`. Net effect: a no-op consuming log space, false sense of coverage in `molecule-full-stack.md §5`. Removing `failed_when: false` lets the command's natural rc=1 surface as a task failure; removing the register removes the dead binding. With this change, full-stack now matches `molecule/database/verify.yml`'s already-hard equivalent assertion.

### T-C2 (root password dedup)

In both `molecule/database/molecule.yml` and `molecule/full-stack/molecule.yml`, the key `database__mysql_root_password` was declared twice in the same `group_vars.all:` mapping. YAML's spec is undefined for duplicate keys in mappings; PyYAML's behaviour (used by Ansible) is last-key-wins. Original behaviour was "the second value takes effect" — verified during R2 by tracing `database/verify.yml` line 26 + `full-stack/verify.yml` line 114, which both use the literal second-value password to authenticate. Removed the first declaration; effective value (and thus verify.yml's hardcoded password reference) unchanged.

### T-C3 (testuser auth probe)

Pre-R2, `database__users: [testuser]` was created by `community.mysql.mysql_user` but never exercised. The Ansible task returns ok if MySQL responded — it does not verify that the GRANT statement actually let `testuser` connect or that `priv: testdb.*:ALL` is effective. The new probe runs `mysql -u testuser -p<pw> --socket=... -e "SELECT 1;" testdb`. Authentication failure → mysql exits non-zero → command task fails. Wrong priv (e.g., user can't query testdb) → same. Uses `database__users[0]` indirection to stay aligned with molecule.yml.

### G3 (Debian 12 platform)

Webserver was clean: existing `_webserver__nginx_packages.Debian: nginx` map (in `roles/webserver/vars/main.yml`) means same package name works on Ubuntu and Debian. New platform stanza is structurally identical to molecule/common's existing `debian12-node`.

Database hit a real obstacle: `roles/database/tasks/install.yml` hardcodes `mysql-server` + `mysql-client`. On Debian 12 those packages don't exist — Bookworm ships MariaDB. Surfaced this to user per workspace §4 "report deviations immediately"; user chose option (b) "Molecule prepare 加 MySQL APT repo" — keep the role unchanged, add the upstream repo only in the molecule test fixture. Added a `raw` task in prepare.yml gated by `inventory_hostname == 'debian12-database'`. The role itself stays family-blind to the maintainer's eye; Debian 12 production users would still need an external repo step (recorded for a future role-refactor workstream).

Trade-off accepted: this is a **test-fixture-only** workaround. It does NOT make the database role work on Debian 12 in production; it makes the role's existing assumptions testable on Debian 12. The role-refactor decision (MariaDB compat? family-mapped packages? both?) is logged for a future workstream.
