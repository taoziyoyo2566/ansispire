## 1. 原始目的

为用户设计的"基于 Ansible 的多服务器管理控制系统"：
- 对照业界主流管理控制系统（AAP、Rundeck、SaltStack、Puppet Enterprise）的领域能力
- 识别当前项目在架构层的**显著缺口**
- 给出**分阶段实施蓝图**，让用户有选择路径的能力

## 2. 本轮关注范围

- 管理控制系统的**领域能力图谱**（15 个核心域）
- 当前项目（含 Round 5）的**差距矩阵**
- **目标架构蓝图**（组件分层、数据流、边界）
- **分阶段路线图**（Round 7-12 或更长，每个 Round 独立可交付）

## 3. 本轮不展开

- 不做任何代码/配置实施
- 不重写 Round 5 的内容（Round 5 的工程改进作为地基保留）
- 不讨论具体技术选型的一级决策（留到各子 Round 单独评审）
- 不涉及商业版 vs 开源的决策（AAP 订阅与否由用户决定）

## 4. 判断标准

- 是否覆盖管理控制系统的全部核心域
- 是否与项目"教学 + 生产可用"双重定位兼容

- 是否保留了**向 AAP 过渡**的能力（不锁死在自建方案）

## 5. 管理控制系统的领域能力图谱

一个成熟的多服务器管理控制系统至少需要覆盖以下 **15 个核心域**：

```
┌─────────────────────────────────────────────────────────────┐
│  1. 控制面 (Control Plane)                                  │
│     - 控制器 / API 网关 / UI / 作业调度                     │
├─────────────────────────────────────────────────────────────┤
│  2. 数据面 (Data Plane / Execution)                         │
│     - Execution Environment / Runner / SSH 执行             │
├─────────────────────────────────────────────────────────────┤
│  3. 身份与访问控制 (IAM)                                    │
│     - RBAC / LDAP/AD/SSO / Team / Organization              │
├─────────────────────────────────────────────────────────────┤
│  4. 库存管理 (Inventory Management)                         │
│     - 静态/动态 / CMDB 对接 / 多环境 / 多租户               │
├─────────────────────────────────────────────────────────────┤
│  5. 密钥与凭据 (Secrets & Credentials)                      │
│     - 分层密钥 / 轮换 / HashiCorp Vault / 云 KMS           │
├─────────────────────────────────────────────────────────────┤
│  6. 可观测性 (Observability)                                │
│     - 集中日志 / 指标 / 分布式追踪 / 告警                   │
├─────────────────────────────────────────────────────────────┤
│  7. 变更管理 (Change Management)                            │
│     - 审批流 / 变更窗口 / 冻结期 / 回滚                     │
├─────────────────────────────────────────────────────────────┤
│  8. 审计与合规 (Audit & Compliance)                         │
│     - 操作日志 / 谁改了什么 / CIS/STIG / 合规扫描           │
├─────────────────────────────────────────────────────────────┤
│  9. 事件驱动 (Event-Driven Automation)                      │
│     - EDA / Webhook / Alertmanager 触发器 / 自愈            │
├─────────────────────────────────────────────────────────────┤
│  10. 多租户与隔离 (Multi-Tenancy)                          │
│      - 环境隔离 / 团队命名空间 / 资源配额                   │
├─────────────────────────────────────────────────────────────┤
│  11. 灾难恢复 (DR)                                         │
│      - 控制器备份 / 库存备份 / 多区域 / 恢复演练            │
├─────────────────────────────────────────────────────────────┤
│  12. 集成层 (Integrations)                                 │
│      - ITSM (ServiceNow/Jira) / 监控 / CI/CD / IM          │
├─────────────────────────────────────────────────────────────┤
│  13. 服务目录与自服务 (Self-Service)                       │
│      - 作业模板 / Survey / 审批工作流 / 目录               │
├─────────────────────────────────────────────────────────────┤
│  14. 扩展性 (Scaling)                                      │
│      - 控制器 HA / 并发执行 / 分布式 Runner / 库存规模     │
├─────────────────────────────────────────────────────────────┤
│  15. 安全加固 (Security Baseline)                          │
│      - CIS Benchmark / STIG / 最小权限 / 网络分段          │
└─────────────────────────────────────────────────────────────┘
```

## 6. 当前项目差距矩阵

> 图例：🟢 完整 / 🟡 部分 / 🔴 缺失 / ⚪ 未启动

| # | 领域 | 当前状态 | Round 5 贡献 | 差距 |
|---|------|---------|-------------|------|
| 1 | 控制面 | 🔴 | 无 | 仅 CLI（ansible-playbook/navigator），无控制器、API、UI、调度 |
| 2 | 数据面 | 🟡 | EE 定义、navigator 配置 | EE 未构建、无 Runner 集群、无分布式执行 |
| 3 | IAM | 🔴 | 无 | 仅 SSH user/become，无 RBAC、无 SSO、无 Team 概念 |
| 4 | 库存管理 | 🟡 | AWS/Azure/GCP 动态 inventory 骨架 | CMDB 未对接、多租户无边界、无库存合并策略 |
| 5 | 密钥 | 🟡 | External SM 示例（注释态） | 未激活、无轮换策略、无分层密钥设计 |
| 6 | 可观测性 | 🔴 | human_log 回调、profile_tasks | 无集中日志、无指标导出、无告警、无审计日志 |
| 7 | 变更管理 | 🔴 | 无 | 无审批、无变更窗口、无冻结期、无回滚策略 |
| 8 | 审计 | 🔴 | .secrets.baseline | 无操作审计、无合规扫描（CIS/STIG） |
| 9 | 事件驱动 | 🟡 | EDA 目录骨架 + README | 无实际规则、无源连接、无事件路由 |
| 10 | 多租户 | 🔴 | production/staging 两环境 | 无团队隔离、无命名空间、无资源配额 |
| 11 | DR | 🔴 | 无 | 无备份策略、无恢复演练、无 RTO/RPO 声明 |
| 12 | 集成层 | 🔴 | 无 | 无 ITSM、无监控对接、无 IM 通知、无 webhook 入口 |
| 13 | 自服务 | 🔴 | 无 | 无作业模板、无 Survey、无服务目录 |
| 14 | 扩展性 | 🔴 | forks=10 | 无 HA 设计、无 Runner 分布式、无库存分片 |
| 15 | 安全加固 | 🟡 | SSH 加固、firewall、accept-new | 无 CIS 对标、无合规配置、无最小权限审计 |

**量化**：15 个核心域中，完整覆盖 **0 个**，部分覆盖 **5 个**，完全缺失 **10 个**。

**结论**：当前项目（含 Round 5）是一个**优秀的 Ansible 内容层骨架**，但**不构成管理控制系统**。Round 5 对 Domain 2、4、5、9、15 提供了"钩子"或"骨架"，其他 10 个域需要从零设计。

### 7.2 关键架构决策

| 决策点 | 选项 | 建议 | 理由 |
|--------|------|------|------|
| 控制器 | AAP（商业）/ AWX（开源）/ 自建 | **AWX（起步） + AAP 兼容** | AWX 是 AAP 上游，结构一致，可后续平滑迁移到订阅版 |
| 密钥存储 | ansible-vault / HashiCorp Vault / 云 KMS | **HashiCorp Vault** | 跨云、支持动态密钥、审计完整 |
| 可观测性 | ELK / Grafana LGTM / 云原生 | **Grafana LGTM 栈** | 开源可本地部署、与 Prometheus 生态一致 |
| EDA | 自建 / ansible-rulebook / AWX EDA | **ansible-rulebook + AWX EDA** | 官方路径、与控制面统一 |
| CMDB 对接 | NetBox / ServiceNow CMDB / 自建 | **NetBox（起步）** | 开源、API 完整、是 Ansible 生态标配 |
| 库存合并 | 多文件静态 / 多 plugin / 统一聚合 | **统一聚合** `constructed` 插件 | 多来源合并到单一视图 |
| 审批流 | 手工 / GitLab MR / AWX Workflow | **AWX Workflow + MR 双轨** | 变更既可代码化（MR）也可运行时（Workflow） |

### 7.3 数据流示例（典型作业）

```
开发者提 MR
    │
    ▼
CI/CD 运行 lint + syntax + molecule + dry-run
    │
    ▼ (合并到 main)
Webhook 触发 AWX 同步
    │
    ▼
用户在 AWX UI 启动作业模板
    │
    ▼
审批工作流（条件：prod 环境 → 需经理审批）
    │
    ▼
AWX 调度器分配 Runner
    │
    ▼
Runner 拉取 EE 镜像 → 执行 playbook
    │
    ▼ (并发分 batch)
被管主机执行
    │
    ▼
结果写入：
  ├─ AWX 活动流（审计）
  ├─ Loki（日志）
  ├─ Prometheus（指标）
  └─ Slack（通知）
```

## 8. 分阶段实施路线图

**总原则**：每个 Round 独立可交付、不影响已交付部分的可运行性。

### Round 7 — 控制面启动（规模：中）

**目标**：用 AWX 搭建最小可用控制器；把现有 playbook/inventory 纳入 AWX 管理。

交付物：
- `docker-compose.awx.yml` 或 `k8s/awx/` 清单
- AWX 初始化 playbook（创建 Organization / Project / Inventory Source / JT）
- 文档：从 CLI 到 AWX 的迁移指南

NFR 关注：正确性、开发者体验

### Round 8 — IAM + 审计（规模：中）

**目标**：RBAC + SSO（OIDC）+ 集中审计日志。

交付物：
- AWX RBAC 设计文档（Org / Team / Role 映射）
- OIDC 集成配置（Keycloak 或 Authentik）
- 审计日志导出到 Loki

NFR 关注：正确性、安全、合规

### Round 9 — 可观测性栈（规模：中）

**目标**：Grafana LGTM 栈部署 + Ansible 执行指标埋点。

交付物：
- Prometheus/Loki/Tempo/Grafana 部署清单
- Ansible 回调插件：执行指标推送到 Prometheus
- Grafana 仪表盘：执行统计、失败率、耗时热力图
- Alertmanager 规则：失败作业告警

NFR 关注：可观测性、可用性

### Round 10 — 密钥管理升级（规模：小）

**目标**：把 ansible-vault 逐步迁移到 HashiCorp Vault；密钥分层（dev/staging/prod）。

交付物：
- HashiCorp Vault 部署清单（或 dev 模式启动脚本）
- AWX Credential Plugin 配置（使用 Vault 作为后端）
- 密钥轮换策略文档
- 迁移脚本：ansible-vault → HashiCorp Vault

NFR 关注：安全

### Round 11 — 事件驱动 + 集成（规模：大）

**目标**：EDA 规则集 + ITSM / 监控 / IM 集成。

交付物：
- AWX EDA 规则：Alertmanager → 作业触发
- ServiceNow / Jira webhook 入口
- Slack / Teams 通知模板
- CMDB 同步（NetBox → AWX Inventory）

NFR 关注：可观测性、可靠性

### Round 12 — 多租户 + DR（规模：大）

**目标**：团队命名空间 + 资源配额 + 备份恢复演练。

交付物：
- AWX 多 Organization 设计
- 资源配额和限流策略
- AWX + Postgres 备份脚本
- DR 演练手册（RTO 30min / RPO 15min）

NFR 关注：可用性、合规

## 9. Round 5 工程地基的价值重估

用户指出 Round 5 "不够完善"，但 Round 5 的价值 **不应被否定** — 它是管理控制系统的**内容层地基**。

| Round 5 产出 | 在管理控制系统中的角色 |
|-------------|---------------------|
| `execution-environment.yml` | 数据面 EE 镜像定义 |
| `.ansible-navigator.yml` | 开发者本地调试工具 |
| `pyproject.toml` | 开发环境声明 |
| `molecule/full-stack/` | 集成测试的基础 |
| `secrets_external.example.yml` | Round 10 HashiCorp Vault 的迁移锚点 |
| `extensions/eda/rulebooks/` | Round 11 EDA 规则集的存放位置 |
| `inventory/dynamic/` 多云 | Round 7 AWX Inventory Source 的数据源 |
| `roles/` 三个 role | 被 AWX 作业模板引用的内容层 |
| `tox.ini`, CI `dry-run` | 作为 AWX 之前的"门禁"保留 |

**结论**：Round 5 = 内容层与开发者体验地基；Round 6+ = 在地基上建造控制系统。

## 10. 风险与边界

### 10.1 主要风险
- **范围爆炸**：15 个域全做=6+ 个月工作量，必须分批
- **技术锁定**：若选了 AWX 后难切 AAP，需验证兼容性
- **运维成本**：引入 Vault/LGTM/EDA 后，运维复杂度显著上升
- **教学价值流失**：控制系统完整后，"学 Ansible 核心"的学习路径被埋没

### 10.2 建议的边界
- **Round 7-9 为必要基线**（控制面 + IAM + 可观测）
- **Round 10-12 为按需增强**（密钥/EDA/多租户）
- 每个 Round 保留 **"纯 CLI 兼容路径"** — 不强制用户走 AWX

## 11. 本文档产生的新发现

以下问题在 Round 5 未被识别，需纳入未来 Round 讨论：

- 当前 `inventory/production` 和 `inventory/staging` 的结构无法支撑多租户
- `ansible.cfg` 的 `log_path` 被注释掉了，集中日志需要重新设计
- 缺少一个统一的 **playbook 入口规范**（哪些是"管理动作"可被 AWX 调用）
- 自定义 callback 插件 `human_log.py` 不适合控制系统（显示导向，不是机器解析导向）
