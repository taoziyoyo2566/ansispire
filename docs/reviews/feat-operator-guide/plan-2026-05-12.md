## 1. Purpose

Bring `docs/user-guide/` to a state where a maintainer **without AI assistance** can execute the full set of day-1 + day-2 operations end-to-end. The "完善的文档" the user asked for is not a greenfield document — `docs/user-guide/` already exists with that exact stated audience ("End-users / operators — long-form, with rationale", per `docs/README.md`) and 5 files. The gap is **completeness**, not existence.

This workstream audits the existing 5 files, identifies what an unassisted operator cannot do today, and fills those gaps.

## 4. Judgment criteria (the "unassisted operator" rubric)

A doc passes the audit if, for every scenario the doc claims to cover, a reasonably-skilled operator (Ansible-fluent, has read the README) can execute it **without** any of:

- (a) Asking an AI for clarification
- (b) Reading source code to figure out a variable / path
- (c) Inferring a command that the doc references but does not spell out
- (d) Cross-referencing 3+ other docs to assemble a procedure
- (e) Following a step whose failure mode is unstated (i.e., the doc must say "if this fails, see X")

Round 2 (the implementation round) will produce an explicit audit table with one row per scenario × rubric criterion.

## 8. Risk / NFR

- **Audience leak**: `operations/*` should stay terse. If the audit finds operations-style content in user-guide (or vice versa), correct in place rather than letting both files grow toward the middle.
