## 一、调查背景

本项目同时被 Claude CLI 和 Gemini CLI 使用。两者各自有独立的规则文件：
- Claude: `CLAUDE.md`（项目级）+ `~/.claude/CLAUDE.md`（用户级）+ `~/workspace/CLAUDE.md`（工作区级）
- Gemini: `GEMINI.md`（项目级）+ `~/.gemini/settings.json`（用户级）

调查目标：Gemini 规则体系是否有优于 Claude 的机制？结论应是「引入部分规则」还是「Gemini 规则覆盖替换 Claude 规则」？

## 二、Gemini 完整配置层次（调查发现）

| 层级 | 文件/路径 | 内容 |
|------|-----------|------|
| 项目级指令 | `./GEMINI.md` | AI 行为规则（4个章节） |

| 用户级设置 | `~/.gemini/settings.json` | 认证方式（oauth-personal）、首选编辑器（vim） |
| 项目注册表 | `~/.gemini/projects.json` | 路径→项目名映射 |
| 信任目录 | `~/.gemini/trustedFolders.json` | `/home/netcup` = DO_NOT_TRUST（安全隔离） |
| 自动记忆 | `~/.gemini/tmp/ansispire/memory/GEMINI.md` | Gemini 自动追加的项目记忆（目前仅2条） |

**关键发现**：Gemini 的配置体系比 `GEMINI.md` 表面上看起来更完整，但用户级设置极简（仅认证+编辑器），自动记忆系统在本项目几乎为空（只有2条原始记录）。

## 三、GEMINI.md 现有内容解析

```
§0 AI Behavioral Protocol
    - Proactive Challenge：变更前必须做影响分析，不盲目执行
    - Sync Guard：任务完成前检查 SUMMARY.md / README.md / Feature Map 是否需同步

§1 Workload Classification & Branch Constraints
    - Task Ledger：TODO.md 存在 todo 分支，用 git show todo:TODO.md 读取
    - Branch Naming：feat/（需Design RFC）、fix/（需RCA）、refactor/（纯等价）

§2 Layered Context Governance（Lazy-loading）
    - 读取顺序：SUMMARY.md → todo:TODO.md → docs/features/*/summary.md → code
    - 目的：防止 AI 上下文溢出，按需加载

§3 Engineering Standards
    - Control vs Data：Controller 与 Role 解耦
    - Vendor Integrity：外部 Role 的本地补丁记录在 SUMMARY.md
    - Evidence-based Verification：每个单元提供 lint/syntax/test 终端日志
    - TSVS：所有测试用 docs/test-specs/TEMPLATE.md 格式记录，无记录不算Done
```

## 四、规则维度对比矩阵

| 维度 | Claude CLAUDE.md | Gemini GEMINI.md | 优胜方 |
|------|-----------------|-----------------|--------|
| 配置层次 | 3层（用户→工作区→项目） | 2层（用户settings + 项目） | **Claude** |
| 任务持久化 | 会话内 TaskCreate（重启消失） | `todo` 分支 TODO.md（跨会话永久） | **Gemini** |
| 上下文加载顺序 | 无明确定义 | 显式4层懒加载序列 | **Gemini** |
| 文档一致性守卫 | 无专项机制 | Sync Guard：完成前3点文档检查 | **Gemini** |
| 分支命名规范 | 无 | feat/fix/refactor + RFC/RCA 要求 | **Gemini** |
| 架构真相文件 | 无强制要求 | SUMMARY.md 作为全局架构 SSOT | **Gemini** |
| 上下文过滤 | 无 .claudeignore | .geminiignore（精确控制读取范围） | **Gemini** |
| 计划优先制度 | 完整（plan+changelog+命名规范） | 较弱（仅证据验证） | **Claude** |
| 自我演化机制 | 4步反馈闭环，CLAUDE.md 自更新 | 无（静态文件，靠手动维护） | **Claude** |
| 级别声明 | Architecture/Engineering/Fix/Exploration | 无级别区分 | **Claude** |
| 范围边界控制 | 明确（禁止未授权扩展，偏差立即上报） | 仅"主动挑战"，不如 Claude 严格 | **Claude** |
| 记忆系统 | 结构化类型（user/feedback/project/reference） | 自动追加无结构（2条原始记录） | **Claude** |
| 成本/限额意识 | §1-C 成本预算，§3 分轮规则 | 无 | **Claude** |
| 积累历史教训 | R1-R12（12条真实失败提炼） | 无 | **Claude** |

统计：**Claude 优胜 8项，Gemini 优胜 6项**

## 五、决策分析

### 方案A：替换（Gemini 规则全面覆盖 Claude）

**否决**。原因：
- CLAUDE.md 有 Gemini 缺少的关键机制：自我演化、级别声明、12条积累智慧、3层配置层次
- GEMINI.md 是静态文件，永远不会从失败中学习；Claude 的反馈闭环是项目真正的长期价值
- Gemini 自动记忆系统在本项目几乎为空，证明其持久化效果远不如 Claude

### 方案B：并列引入（两套规则独立共存）

**否决**。原因：
- 双套治理 = 规则冲突时 AI 无法判断权威来源
- 同一项目两个 AI 应共享规则内核，差异只在各自的工具调用方式上
- 维护成本翻倍

### 方案C：选择性集成（推荐）

将 Gemini 中 6 个优于 Claude 的机制**吸收进 CLAUDE.md**，GEMINI.md 精简为指向共享规则的薄层。两个 AI 共享同一套项目治理标准，差异只在 AI 工具能力层面。

### 文件变更清单

| 文件 | 变更类型 | 摘要 |
|------|---------|------|
| `CLAUDE.md` | 修改 | 吸收 G1~G5，添加 R13~R17，§1/§2/§3/§8 各节扩充 |

### 明确不做的事

- 不删除 `.geminiignore`（它有独立价值，控制 Gemini 上下文读取范围）
- 不合并两个 AI 的记忆系统（各自独立，运行机制不同）
- 不改变 `~/.gemini/settings.json`（用户级设置与项目治理无关）
- 不修改 `~/.claude/` 或 `~/workspace/CLAUDE.md`（本轮仅限项目级）
