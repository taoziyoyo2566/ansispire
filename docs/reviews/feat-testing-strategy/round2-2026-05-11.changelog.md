## 3. 每项改动的意图

### 3.1 INDEX.md

**为什么需要 INDEX**：项目此前 6 份 TSVS 平铺在 `docs/reference/test-specs/`，
新加入者需要 `ls` + 逐文件读才能盘点覆盖面。本 workstream 又新增 4 份后，
不建索引会让"哪个表面有规格 / 哪个没有"的判断成本继续上升。

**结构选择**：
- §1 注册表 = 单一信源；列：ID / 首次登记 / 表面 / 层级 / Carrier / 状态 / 链接。
- §2 命名约定显式收口"新 TSVS 用 `TSVS-<SCOPE>-<SLUG>-NNN`"，避免后续随机化。
  历史 `TEST-EDA-*` ID 不重命名（迁移无收益）。
- §3 表面覆盖映射 = 反向查找表，回答"我改了 X 表面，有谁覆盖了它"。
  与 `test-plan.md §2` 互为镜像（前者 surface-first，后者 TSVS-first）。
- §4 维护流程明确"落地 / 退役 / 状态迁移"三种操作的执行步骤。
- §5 变更日志倒序，未来每次 INDEX 自身变化都在此追加一行。

### 3.2 4 份 Molecule TSVS

**为什么**：plan §3 已识别"4 个 Molecule 场景断言意图隐式，contributor 必须读 verify.yml 才知验什么"。
本轮的目标是**登记现状**，不是改进；任何新断言都属于下一个 workstream。

**结构选择**：
- 严格按 `TEMPLATE.md` 7 章顺序展开，与现有 6 份 TSVS 风格对齐。
- §5 预期结果以 `verify.yml` 实际任务为锚（标注硬断言 vs soft check），
  避免凭印象描述；full-stack §5 显式拆为 common 子集 / webserver / database / coexist 四块。
- §5.1 显式登记"未断言项"，让覆盖盲区可见（与 `test-plan.md §5` 缺口列表交叉引用）。
- §6 实际执行记录留为"由 CI 持续维护，本 round 未独立运行"——
  避免在文档中编造未发生的执行结果。
- §7 结论与建议每份都包含"维度扩展 / 现有盲区 / 修复回归路径"三类后续工作建议。

### 3.3 Makefile help 文本

**为什么**：`make help` 是 contributor 首次接触 verify 三件套的入口；
原文本只说"包含什么"，没说"何时用"。本次最小修改：仅在每条 `## ...` 注释中加
"Save-point / Push / Release gate"语义 + 耗时区间。

**未改动**：target 依赖、命令体、shell 行——本轮仍是 doc-only 精神。

### 3.4 治理文档 cheatsheet

**为什么**：`testing-governance.md` 是 165 行长文；新人从头读到 §3 决策树需要 1–2 分钟。
顶部一行 cheatsheet 把"verify-quick → verify → verify-full"的递进时机一句话讲清，
让 contributor 在不读全文的情况下也能做正确决策。决策树仍是真理之源，
cheatsheet 只是快速入口。

### 3.5 test-plan.md G4/G5 闭合 + TSVS 列填充

**为什么**：plan §5 缺口列表是"真相清单"——既然本轮真的关上了 G4/G5，
列表必须立刻同步，否则下一轮会再次把这两项当成 open work。
用 `~~strikethrough~~` + `✅ CLOSED 日期` 表达"历史曾有这个缺口，已闭合"，
保留历史痕迹而不是删除原行。
