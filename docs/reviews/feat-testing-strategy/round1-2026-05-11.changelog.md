## 3. 每项改动的意图

### 3.1 `testing-governance.md` 重写

**原因**：HEAD 版本只有 3 节、20 行，定义了"环境一致性""全链路闭环""手动 syntax 截图"等
模糊指令，无法回答 plan §4 列出的具体痛点："改了 X 文件应该跑什么"。

**结构选择**：
- §2 测试金字塔显式登记每层**现项目载体**（具体到 `controller/audit/test_reactor.py` 14 cases），
  避免抽象层级名失锚；
- §3 决策树 16 行路径覆盖项目所有 first-party surface，外加 Makefile / `.github/workflows` /
  `docs/governance/**` 这些"meta-改动"应触发的同步动作；
- §4.2 显式标注"本地 `molecule-all` 串行 / CI matrix 并行"——回应 round 5 中"测得太慢"
  的真实根因（本地痛点 ≠ CI 痛点）；
- §6 Molecule 双模式表（`test` vs `converge`）解释"为什么有时候要 destroy，有时候不要"，
  让贡献者在迭代阶段不再被高频环境重建惩罚；
- §7 TSVS 强制规约把 plan §3 中已识别的"4 个 Molecule 场景无 TSVS"缺口转成红线（后续 round 2 收口）。

### 3.2 `test-plan.md` 新建

**原因**：项目没有任何一份文档**单独**回答"我们到底测了什么"。CI 状态绿不等于覆盖充分；
plan §4 的痛点根因（"L0–L3 物理上抓不到部署期 bug，必须 L4"）需要矩阵化呈现才能让
未来 contributor / 审计员理解。

**结构选择**：
- §2 表面清单 = 一行一个第一方代码 / 配置区，附 TSVS 文档列（缺则 round 2 补，红线明确）；
- §3 覆盖矩阵刻意把"防火墙 + intra-host""模板渲染产物""二次执行幂等"三行加粗 ——
  这正是 round 5 五个 bug 全部踩中的类型，并且全部只能由 L4 浮现；
- §4 把每个 `verify.yml` 的断言清单逐条登记，让"已测"不再凭印象；
- §5 缺口表分 risk × 责任轮次，避免"看到一堆缺口不知道哪个先做"——
  G4/G5 标 round 2（本 workstream 内），G1/G2/G3/G6/G7 标 Tier C（独立工程改造）。

### 3.3 CHANGELOG 同步

**为什么写**：contributor-facing 治理规则变化（TSVS 红线、决策树）虽然不属 §0 Sync Guard
列举的 user-visible 类型，但属于该文件其他类似条目（如已有的 "Documentation refactor"）的
延续，让 CHANGELOG 仍是"近期发生了什么变化"的单一可读入口。
