## 1. Purpose / Scope / Level

**Purpose**: replace the current 20-line `docs/governance/testing-governance.md`
(thin process rules) with a real **testing strategy** (the "why / how to choose
what to run when") and a **test plan** (the "what is tested / what is not
tested / coverage matrix"). Address efficiency pain points the user raised
this round.

**Out of scope** (this workstream, even at largest tier):
- Replacing existing test code logic (no rewriting `controller/audit/test_*.py`,
  no rewriting `molecule/*/verify.yml` assertions). We may *reorganize* and
  *add new tests*, not rewrite working ones.
- Migrating to a different test framework (no swap from Molecule to anything
  else, no swap from pytest to unittest, etc.).
- Performance benchmarking infrastructure.

**Level**: per project §1, this is **L1.5 Investigation** (current-state
diagnosis) + **L2 Architecture** (proposing the test pyramid structure +
quality gate decisions across the project surface). Per workspace §1-A this
warrants the full Plan-First treatment before any implementation.

## 2. NFR priorities (APPROVED order)

In decreasing priority:
1. **Correctness gate confidence** — no regression slips through `make verify`.
   Today: lint + syntax catch pure-text issues; EDA pyramid covers the
   reactor/controller; Molecule covers infra. **Rounds 5+6 surfaced that
   `lint + syntax + dry-run` did NOT catch UFW-lo, `ansible_managed`
   un-commented, MySQL idempotency, vhost variable divergence — only
   Molecule did.** This is the single biggest gap.
2. **Local feedback latency** — `make verify-quick` < 5 s, `make verify` < 60 s,
   `make verify-full` (with Molecule) < 5 min. Currently `verify-full` is
   serial-Molecule and likely > 10 min on any change.
3. **Test discoverability** — a contributor must be able to answer "what
   should I run after editing `roles/database/`?" in under 30 s, by
   reading one document.
4. **Coverage completeness** — every first-party role and controller
   subsystem has at least one TSVS-tracked spec.

## 4. Pain-point root cause (user's diagnosis + my structural reading)

User's three causes (round-5 retrospective):
1. **Hidden bugs surfaced late** → because the only layer that could have
   caught them (Molecule) is not in the default `verify` chain.
2. **Container rebuild per fix** → Molecule's `test` lifecycle destroys +
   recreates between iterations. Use `molecule converge` (preserves the
   instance) for iterative development; reserve `molecule test` for the
   final clean-room pass.
3. **Serial execution** → `make molecule-all` is a serial shell loop.
   GNU make `-j` won't parallelize it as written; need explicit refactor.

Structural causes I add:
4. No TSVS docs for the four Molecule scenarios — meaning the assertion
   intent is undocumented and contributors guess what to verify.
5. Three first-party roles (`ansispire_hub`, `ansispire_audit`,
   `infra_baseline`) have no Molecule-level scenario — they get only
   lint+syntax (and for audit, EDA pyramid). A bug there is invisible
   to `verify-full`.
6. `make verify` and `make verify-full` are two binary modes; there is
   no middle tier "lint + syntax + ONE molecule scenario based on what
   I touched" — every contributor is forced to choose between cheap-and-blind
   or thorough-and-slow.

## 5. Decisions (recorded 2026-05-10)

### 5.1 Scope — Tier B

Three tiers were considered:

| Tier | What lands | Estimate | Decision |
| :--- | :--- | :--- | :--- |
| A — docs only | 2 governance docs, no Make/CI/test-spec changes | 1 round (~2–3 h) | rejected — leaves discoverability gap (test-specs/ stays unorganized, no Molecule TSVS) |
| **B — docs + restructure** | A + reorganize `docs/reference/test-specs/` into a discoverable INDEX + add TSVS docs for the 4 Molecule scenarios + clarify Make targets. **No test-code changes.** | 2 rounds (~4–6 h) | **CHOSEN** |
| C — docs + restructure + verify-chain refactor | B + new `make verify-mid` tier (lint+syntax+single-scenario Molecule); parallel `molecule-all`; molecule-converge dev pattern documented. May add Molecule scenarios for `ansispire_hub` / `ansispire_audit` / `infra_baseline`. | 3–4 rounds (~10–15 h) | deferred to a future workstream — wait until Tier B's strategy doc tells us where to invest |

**Why B over A**: A produces docs that immediately drift because they
describe a structure that does not exist (no Molecule TSVS to point at, no
test-specs index to navigate). B fixes the substrate the docs describe.

**Why B over C**: C bundles physical-infrastructure decisions (parallelization
strategy, scenario sharding rules, Molecule for 3 more roles) with
strategy-writing. Doing both at once means strategy choices get pre-empted
by what's easy to implement. Cleaner to write the strategy first, then let
strategy drive infra investment.

### 5.2 Document structure — Two files

| File | Role | Audience |
| :--- | :--- | :--- |
| `docs/governance/testing-governance.md` (rewrite) | **测试方针** — pyramid, decision tree, operating modes, gate semantics | Contributor + Maintainer + Future agent |
| `docs/governance/test-plan.md` (new) | **测试计划书** — per-surface coverage matrix, assertion inventory, gap classification, acceptance criteria | Auditor + Maintainer + Future agent |

Single-stratified-file alternative was rejected: longer file, mixes
"how we test" (stable) with "current coverage state" (drifts) — separate
files let each evolve at its own cadence.
