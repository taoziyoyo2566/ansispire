## 2. Intent per Change

### WU-1.1 — `controller/semaphore/docker-compose.yml` (DB_PATH)
**问题**：原值 `SEMAPHORE_DB_PATH: /var/lib/semaphore/semaphore.db` 与上游 `deployment/docker/server/server-wrapper` 语义冲突 —— wrapper 把该值当作**目录**，自动追加 `/database.sqlite`，导致 `semaphore.db` 作为目录被创建。功能勉强可用但语义错乱，且操作员困惑。
**意图**：删除显式赋值，依赖上游默认 `/var/lib/semaphore`。这与 compose 的 `volumes` 段 `semaphore-data:/var/lib/semaphore` 一致。加注释固化"该变量是目录而非文件"的知识。

### WU-1.2 — `roles/ansispire_hub/tasks/main.yml` (token first-deploy)
**问题**：首次部署时 `stat /var/lib/ansispire/state/.eda_token` 返回 exists=false → mint block 运行并写文件 → 下方 `slurp` 任务的 `when: ansispire_hub_token_file.stat.exists` 仍用 stale 值 false → 跳过 → `ansispire_hub_eda_token` fact 永不设置 → 下游 `ansispire_audit` 启 reactor 时拿到空 `SEMAPHORE_API_TOKEN`。
**意图**：在 mint block 末尾（写 `.eda_token` 后）直接 `set_fact ansispire_hub_eda_token: "{{ ansispire_hub_token_out.json.id }}"`，确保首次部署链路打通。原 slurp + set_fact 路径保留以服务"再部署"路径。两条路径用任务名 `(first-deploy path)` / `(re-deploy path)` 区分。

### WU-1.3 — `controller/audit/reactor.py` (webhook impl)
**问题**：rules.json 的 `Remediation: DB Connection Failure` 规则带一个 `webhook` action（`"DB Alert"`），但 reactor 代码里该 action type 的 dispatch 是 `pass`，rule 启用后**静默**失败。
**意图**：实现 `trigger_webhook(action, event_payload)`：POST 一个 JSON `{rule_action, event}` 到 `action.url`，10s timeout，空/缺失 url 走 skip log（不报错），网络异常被 catch 不抛出。这样 webhook action 真正可用，且失败模式有可观察日志。

### WU-1.4 — `roles/ansispire_hub/tasks/main.yml` (rsync runtime/)
**问题**：rsync excludes 列表覆盖 `.git/.venv/.claude/.ansible/__pycache__` 等本地工件，但漏掉 `runtime/`。`runtime/` 是 vps_manager 的工作站本地 inbox/logs/state 目录（`.gitignore` 已排除但 rsync excludes 没排除）→ 每次 `make hub-deploy` 会把工作站的 VPS 处理状态推到 hub。虽非密文但意外。
**意图**：在"Local-only artefacts"段末尾加 `--exclude=runtime/`，与 `.gitignore` 同步。
