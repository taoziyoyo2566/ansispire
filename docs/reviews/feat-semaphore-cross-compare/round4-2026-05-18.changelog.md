## 3. Intent per Change

### 3.1 bootstrap.yml git_branch (2.1)
原 hardcoded `git_branch: "master"` 在 dev-trunk 阶段是 wrong default。改为 `{{ semaphore_git_branch }}` + `semaphore_workspace_git_branch | default('dev')`，保留 override 通道。注释说明因为 `/workspace` 是 Local repo bind-mount，git_branch 大多数情况下只是元数据。

### 3.2 reactor mtime cache (2.2)
原 `load_rules` 每 poll tick 都 open+json.loads — 1s poll 下每秒一次 disk hit + 解析。新版本：
- 缓存 `mtime / loaded_at / rules`
- 文件不存在 → 返回缓存（防止短暂 missing 触发 reset 风暴）
- mtime 未变 + `(now - loaded_at) < 30s` → 直接返回缓存
- mtime 未变 + 超过窗口 → 只刷新 loaded_at（避免无意义 stat）
- mtime 变了 → 真重读 + 日志记录新规则数

结果：busy editor / `inotify` storm 下 reactor 不会被 reload 风暴卷死；同时还保留 hot-reload 能力（新规则 30s 内生效）。

### 3.3 reactor 外层 while-True (2.3)
原 `main()` 在 fatal exception 后递归 call `main()` — 长期运行后调用栈每次错误增长 8KB；当系统进入 "JSONL 文件被 logrotate 处理" 周期时容易耗内存。

新版本：
- `run_tail_loop()` 是单次 pass
- `main()` 是 `while True: try: run_tail_loop(); except KeyboardInterrupt: return; except Exception: log + sleep`
- 调用栈深度始终是 2 层不变

### 3.4 reactor cursor 持久化 (2.4)
原 `main()` 一上来就 `f.seek(0, os.SEEK_END)` — 重启即丢失所有未消费事件。新版：
- `load_cursor()` 从 `CURSOR_FILE` 读 offset，0 / -1 / 不存在 → 走 EOF 路径
- `save_cursor(offset)` 用 `.tmp` swap 实现 atomic-ish 写入
- 每 5s（CURSOR_FLUSH_INTERVAL）刷新；idle 时 + 处理事件后都有机会触发
- truncation 检测：cursor > file size → seek EOF（logrotate copytruncate 场景）

### 3.5–3.6 Dockerfile baked-in (2.5/2.6)
原 compose `command:` 用 `apk add --no-cache logrotate >/dev/null && ...` 启动每个容器。问题：
- 每次 container start 都跑 `apk update` + install — slow on cold cache
- alpine CDN 不可达时 silent fail，container exit code 0（启动脚本继续执行了）
- 实际运行 image ≠ pinned base image（运行后 apk db 状态被修改）

新 Dockerfile：
- baked logrotate（sink）+ jq + procps（reactor）
- COPY sink.py / reactor.py 进 image，compose 不再 bind-mount 这两个脚本（脚本即 image 的一部分，rebuild 是 promotion 入口）
- image tag 加 `ansispire/` 前缀避免和上游 `python:3.12-alpine` 混淆

### 3.7 compose healthcheck 扩展 (2.7)
原 compose 只 sink 有 healthcheck；relay + reactor 无 → "running" ≠ "alive"。新加：
- relay：`CMD-SHELL pidof python3 >/dev/null`（busybox 自带）
- reactor：`CMD-SHELL pgrep -f reactor.py >/dev/null`（procps 烘焙）

这两个轻量探针不引入新依赖；30s interval / 3s timeout / 5 retries / 10s start_period，与现有 sink healthcheck 同步。

### 3.8 admin password rotation (2.8)
**这是修复一个潜在的 silent failure**：原 `Hub | Provision Admin & Token` 块被 `when: not token_file.exists` 整体 gate，意味着：
- 首次部署：`user add` 创建 admin（success）
- 二次部署：`user add` 走 UNIQUE 路径被吞，**密码再也不会变**
- 用户在 vault.yml 改密码 + re-deploy → vault 改了，hub 没变；下一次 IaC login 报 401

新结构：把用户管理拆出独立块 `Hub | Enforce Admin User & Password`，**每次 deploy 都跑**：
1. `user add` 用 UNIQUE 吞掉重复（保持幂等）
2. `user change-by-login` 强制写入当前 vault 密码（每次跑，确保密码与 vault 同步）

Token mint 块保留 first-deploy gate（token 是一次性资源）。新增 `no_log: true` 到所有打 password 的 API 任务。
