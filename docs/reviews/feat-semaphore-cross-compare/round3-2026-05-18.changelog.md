## 3. Intent per Change

### 3.1–3.3 三个 security envs 链路
**目的**：吸收上游 OSS 安全建议（[semaphoreui.com/docs/.../security](https://semaphoreui.com/docs/administration-guide/security/)）。空 envs 时 Semaphore 每次重启生成 ephemeral 密钥 → cookie 失效 + AccessKey 解密失败。Path A 通过 hub role 自动生成 state file 并渲染到 .env；Path B 由用户手动设置。

### 3.4 state/.security_keys 生命周期
**设计**：与现有 `.eda_token` 同构（stat → mint-if-missing → slurp → set_fact）。Mint 任务用 bash `set -eo pipefail` + `umask 077` 保证 0600 权限。Check mode 在 fresh host 上设 placeholder fact 使 dry-run 不被 `template undefined` 中断。

**关键安全约束**（已写进 hub-deployment.md + iac-vs-ui-boundary.md）：删除 `.security_keys` 意味着所有已存储的 Semaphore AccessKey 无法解密 + 所有 session 失效，**不可手动重新生成**，只能 restore from backup。

### 3.5 rules.schema.json
**设计**：与 `events.schema.json` 同模式（Draft-07 + `$id` + `version`）。actions 用 `oneOf [action_semaphore_api, action_webhook]` 分支，并通过 `anyOf [project_id, project_name]` 强制 semaphore_api 至少给出一种 project 标识。Negative test 覆盖 5 类违法形态（missing name / webhook 无 url / semaphore_api 无 project / 空 condition / 未知 type）—— 全部正确拒绝（见 §5 Gate 3）。

### 3.6 Makefile test-rules-schema
**实现**：inline Python（`python -c "..."`）调用 `jsonschema.validate` + `Draft7Validator.check_schema`。无新脚本文件，无新 wrapper。加入 `test-eda` 链使现有 `make verify` 自动覆盖。

### 3.7 iac-vs-ui-boundary.md
**目的**：在 cross-compare IVG §5.B "IaC vs UI 边界文档"  findings 中点名缺失，本 round 闭环。Ownership map 覆盖所有 bootstrap.yml 创建的资源类型（project / user / inventory / repo / env / key / template / token），加上未来 runner 的预留行。3 类 hybrid exceptions 显式列出（key placeholder / admin user / sessions）。
